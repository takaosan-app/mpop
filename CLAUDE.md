## 1. コンセプト

### アプリ名
**My Park Our Parks**

### コアアイデア
「自分の好きな公園を記録し、みんなの公園体験を地図で見つける。」

My Park Our Parks は、投稿対象を **公園・緑地・広場・遊歩道などの Park に限定**した記録・共有アプリ。以前の My Spot Our Spot 構想のような汎用的な発見地図ではなく、公園に関する体験、写真、設備、季節の様子、混雑感、子ども連れ・散歩・休憩・運動などの利用文脈に集中する。

本アプリの価値は、公式情報だけでは分かりにくい「その公園が今どんな感じか」「どんな過ごし方に向いているか」「また行きたいか」を、個人記録とコミュニティ共有の両方で扱えることにある。

### 2つのモード

| モード | 説明 |
|---|---|
| **My Park** | 個人記録。自分だけの公園メモ、訪問履歴、写真、気づき、ブックマーク。 |
| **Our Parks** | コミュニティ共有。みんなの投稿を地図やフィードで見て、行きたい公園を見つける。 |

この2モードはUIと情報設計の軸であり、投稿単位で公開範囲を切り替えられる。

## 2. 対象・対象外

### 対象にする Park

- 都市公園、児童公園、近隣公園、総合公園
- 自然公園、緑地、河川敷、遊歩道、散策路
- 広場、芝生エリア、親水空間、展望スペース
- 公園内の設備、遊具、ベンチ、トイレ、水飲み場、日陰、駐輪場
- 公園内や周辺で見られる植物、花、野鳥、昆虫、季節の風景
- 公園での過ごし方に関する記録（散歩、休憩、子ども連れ、ピクニック、運動など）

### 対象外

- 飲食店、商業施設、観光施設そのものを主対象にした投稿
- 個人宅、学校、保育施設、医療施設などプライバシーリスクが高い場所
- 公園と関係のない路上発見、建築、看板、一般店舗レビュー
- 立入禁止区域、危険区域、管理者のルールに反する場所
- 人物を主対象にした写真や、個人を特定できる投稿

### 公園内施設の扱い

公園内のカフェ、売店、駐車場、トイレなどは投稿対象に含める。ただし、Google Maps 等の施設情報と競合する実用データベースにはしない。

| 設計方針 | 内容 |
|---|---|
| 構造化フィールドを限定 | 営業時間、料金、在庫、予約可否などの保証が必要な項目は持たない |
| 投稿の鮮度を常時表示 | 「3日前」「2か月前」など、投稿時点の情報であることを明示 |
| ユーザー投稿であることを明示 | 公式情報に見えるバッジや表現を避ける |
| 通報・修正導線 | 閉鎖、誤情報、不適切写真などを報告できる |
| 免責 | 公園・施設情報の正確性をプラットフォームが保証しないことを表示 |

## 3. 投稿カテゴリ

投稿は「公園をどう記録するか」に合わせて分類する。

| カテゴリ | 内容 |
|---|---|
| **Park Note** | 公園全体の印象、雰囲気、過ごしやすさ、再訪メモ |
| **Play** | 遊具、子ども連れ、広場、ボール遊び可否など |
| **Nature** | 花、木、野鳥、昆虫、紅葉、季節の風景 |
| **Rest** | ベンチ、日陰、静かさ、読書・休憩向きの場所 |
| **Walk / Run** | 散歩道、ランニング、坂、距離感、舗装状態 |
| **Facility** | トイレ、水飲み場、駐輪場、駐車場、売店など |
| **Access** | 駅・バス停からの行きやすさ、入口、段差、ベビーカー可否 |
| **Safety / Caution** | 暗さ、ぬかるみ、工事、混雑、注意点 |

## 4. AI機能

### 4-1. AIモデレーション（投稿前に実行・必須）

投稿フローの最初のステップ。モデレーションを通過した写真だけが分類・保存に進む。

| 判定対象 | 処理 |
|---|---|
| 顔・人物が大きく写っている | 自動ブラー処理、または投稿者に差し替えを促す |
| 子どもが識別可能に写っている | 原則ブラーまたはアップロード拒否 |
| 車のナンバー、住所、表札等 | 検出して警告・修正を促す |
| 猥褻、暴力、危険行為、迷惑行為 | 信頼スコアが閾値超で却下 |
| 判定が曖昧 | 投稿者確認またはレビューキューへ |

**採用候補**: Google Cloud Vision API（SafeSearch + Face Detection）

### 4-2. AI自動分類（モデレーション通過後）

写真とコメントから、公園投稿としてのカテゴリ、タグ、説明候補を生成する。

| 入力 | 出力 |
|---|---|
| 公園全体の写真 | 雰囲気、混雑感、利用シーン、季節タグ |
| 遊具・広場 | Playカテゴリ、対象年齢の推定、注意点 |
| 植物・花・野鳥 | Natureカテゴリ、名称候補、季節情報 |
| 設備写真 | Facilityカテゴリ、設備種別、状態メモ |
| 入口・道・坂 | Access / Walkカテゴリ、段差・舗装・歩きやすさ |

**採用候補**: Claude Haiku 系 Vision モデル、または同等の低コストVisionモデル。

> AIの役割は「公式情報の確定」ではなく、投稿者の入力を軽くする補助。AI生成内容は投稿者が確認・編集してから保存する。

### 4-3. コスト方針

- アップロード時に画像を長辺1,024px以下に圧縮
- サムネイルを別生成し、地図ピン・一覧ではサムネイルを使用
- AI分類は投稿時のみ実行し、一覧表示では再実行しない
- 無料プランの月間投稿数を制限し、AI従量費を制御する

## 5. 投稿フロー

```
写真を撮る / 選ぶ
    ↓
位置情報を取得、または公園を検索して選択
    ↓
AIモデレーション（顔・有害・プライバシー判定）
    ↓
AI自動分類（カテゴリ・タグ・説明候補）
    ↓
投稿者が確認・編集（コメント、公開範囲、場所精度）
    ↓
DB保存（公園、座標、分類、信頼スコア、タイムスタンプ）
    ↓
My Park または Our Parks に表示
```

## 6. 公開設定

投稿ごとに3段階から選択できる。

| 設定 | 地図表示 | 閲覧者 |
|---|---|---|
| **非公開** | 公開地図には表示されない | 自分のみ（My Park） |
| **フォロワーのみ** | フォロワーの地図・フィードに表示 | フォロワー |
| **全体公開** | Our Parks に表示 | 全ユーザー |

### 位置情報の精度設定

公園内の詳細地点を共有する場合でも、プライバシーや希少生物保護に配慮する。

- **公園単位**: 公園の代表地点に丸める
- **おおまかな場所**: 公園内のエリア程度に丸める
- **正確な場所**: GPS座標そのまま

希少種・営巣地・個人宅近接などは、正確な場所での公開を制限する。

## 7. 地図・発見機能

### Our Parks マップ

- 公園単位のピン表示
- 投稿数、直近投稿、カテゴリ傾向を公園カードに表示
- ズームアウト時はエリア単位でクラスタリング
- カテゴリフィルター（Play / Nature / Rest / Facility など）
- 季節フィルター（桜、紅葉、花、日陰、水遊びなど）
- 公園詳細では、写真タイムライン・設備メモ・ユーザーコメントを表示

### 近くの公園を探す

- 現在地から半径 500m / 1km / 5km
- 徒歩・自転車の所要時間目安
- 子ども連れ、休憩、散歩、自然観察などの目的別フィルター
- 「近くの公園の新着」通知オプション

### My Park

- 行った公園の履歴
- 行きたい公園のブックマーク
- 自分だけの写真・メモ
- 再訪したい季節、混雑しにくい時間帯などの個人メモ

## 9. 技術スタック

### 採用候補

```
┌─────────────────────────────────┐
│ フロントエンド / アプリ           │
│ Flutter（iOS / Android / Web）   │
└────────────┬────────────────────┘
             │
┌────────────▼────────────────────┐
│ バックエンド / BaaS               │
│ Supabase                         │
│ ├── PostgreSQL + PostGIS（DB）   │
│ ├── Authentication（認証）        │
│ ├── Realtime（通知・フィード）    │
│ └── Edge Functions（API）        │
└────────────┬────────────────────┘
             │
┌────────────▼────────────────────┐
│ 外部サービス                      │
│ ├── Cloudflare R2（画像保存）    │
│ ├── Google Cloud Vision（モデレーション）│
│ ├── Vision AI（分類・タグ生成）   │
│ └── Google Maps（地図表示）      │
└─────────────────────────────────┘
```

| レイヤー | 採用候補 | 理由 |
|---|---|---|
| アプリ | Flutter | iOS・Android・Webを単一コードベースでカバー |
| DB | Supabase PostgreSQL + PostGIS | 公園検索、近隣検索、地理空間クエリに向く |
| 認証 | Supabase Auth | Google / Apple ログインに対応 |
| 画像ストレージ | Cloudflare R2 | 転送料を抑えやすく、S3互換 |
| AIモデレーション | Google Cloud Vision | SafeSearch + Face Detection |
| AI分類 | Claude Haiku 系 Vision モデル | 低コストで画像説明・分類に対応 |
| 地図 | Google Maps Flutter plugin | 日本国内の地図・住所精度が高い |
| プッシュ通知 | Firebase Cloud Messaging | Flutter対応の標準的な選択 |

## 11. モデレーション運用

### 自動

- 投稿前にリアルタイムで実行
- 判定結果と信頼スコアをDBに記録
- 子どもや第三者の顔が写る写真は厳しめに扱う

### 半自動

- AIスコアが閾値付近の投稿をレビューキューへ
- 通報を受けた投稿もレビューキューへ

### コミュニティ

- ユーザーによる通報
- 一定数の通報で自動的に非表示にしてレビュー待ち
- 公園管理者や自治体からの削除依頼に対応できる導線を用意

## 12. 認証設計

### 採用方式

Google / Apple ソーシャルログインを基本とする。

| 項目 | 内容 |
|---|---|
| 対応プラットフォーム | Googleログイン、Appleログイン |
| アプリが受け取る情報 | OAuthプロバイダー発行のID、必要最小限のメールアドレス |
| 表示名 | ユーザーが自由設定するニックネーム |
| アバター | 任意 |

氏名、住所、電話番号、生年月日は取得しない。
